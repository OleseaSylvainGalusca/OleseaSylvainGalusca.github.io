Streaming Finance Data

AWS Lambda allows creating functions using Python to gather data and transform a data frame from yfinance module into JSON. We can set a trigger for how often we want the data to update, for example, every minute. On trigger, data is delivered to our data source (AWS S3) through a delivery stream (AWS Kinesis). Glue is a managed ETL service provided by AWS. The advantage of using Glue is no need to spin up any infrastructure just point to the location of the data (AWS S3) and optionally its schema. If no schema was provided, we can use a Crawler – a program that connects to the dataset and generates metadata and defines schema. Finally, AWS Athena can be used to query the data interactively using SQL.

Creating a Logistic Model

Predicting Work Absenteeism

Why Logistic Regression? In many decision-making problems, we want to have a more informative prediction than just a classification. For example, rather than simply predicting whether a person will call in sick within a certain period, we might want to have an estimate of the probability that they will call in sick. The most common procedure by which we estimate the probability that a new instance belongs to the class of interest is called logistic regression. This analysis focuses on creating a logistic model that will predict the probability of a person calling in sick. See the original data here.

Check Code on GitHub

Data was pre-processed and loaded for analysis in Jupyter notebook. The data includes observations about 4 coded reasons for absence: 1 - various diseases, 2 - pregnancies, 3 - poisoning related, 4 - medical follow-ups.

To process further our data we will need to create two classes of people who: were moderately absent and those that were excessively absent. We will use a methodology that is quite simple, we will calculate the median value of the 'Absenteeism Time in Hours' and use it as a cut-off line. Everything below the median is normal, everything above will be considered excessive. What are the classes? Moderately absent (<= 3 hours) and Excessively absent (=> 4 hours). If an observation is absent for less than 3 hours then we assign it a zero. If an observation is absent for more than 3 hours then we assign it a 1. In supervised learning, we call these zeros and 1s: TARGETS, because these are the values we are aiming for. A comment on target: using the median as a cut-off line is 'numerically stable and rigid', that's because using the median we have implicitly balanced the dataset. This will prevent our model from learning to output only 0s or only 1s. To prove that, let's divide the number of targets that are 1 by the total number of targets. The total number of targets that are 1 can be found by summing up all values of targets, while all number of targets is the shape on axis 0 (vertical line). A new column will appear in our data set called Excessive Absenteeism which is our TARGETS.

We will create a StandardScaler object (which is empty) that will help us to scale our data. Absenteeism_scaler is a StandardScaler object which will be used to subtract the mean and divide by the standard deviation variablewise (featurewise). To apply the scaling mechanism we need to use another method called transform. This operation transforms unscaled inputs from the absenteeism_scaler. In other words: we subtracted the mean and divided by standard deviation.

Train-Test Split: overfitting means that a model learns to predict the train data so well that it fails miserably on new data. One way to deal with overfitting is to hide a small portion of the data from the algorithm. We train the data on most of it but test a portion. We will split the data set into train and test, to assess the accuracy of the model with data it never seen before. Data shuffle - we want to shuffle the data to remove all the dependencies that comes from the order of the data set (to make it random).

Training model and assessing its accuracy.

Finding the intercept and coefficient: if its coefficient is around zero or if its odds ratio is around 1 the corresponding feature has little importance. A weight (coefficient) of zero implies that no matter the feature value, we will multiply it by zero (in the model) and the whole result will be equal to zero. For a unit change in the standardized feature, the odds increase by a multiple equal to the odds ratio ( 1 = no change). For example if we say: odds * odds ratio = new odds ( for a unit change 5:1 * 2 = 10:1 | 5:1 * 0.2 = 1:1 | 5:1 * 1 = 5:1). The 'Daily Work Load Average' weight (coefficient) is almost zero and odds are almost 1, so this feature makes no difference. So, it’s ‘Day of Week' and 'Distance to Work'.

Testing the model with the rest on the data and assess it's accuracy. As we see below it accounts for 74% accuracy.

Business Intelligence Solution Design

Use case: Businesses need a way to track KPIs (sales, revenue, profit, returns), compare regional performance, analyze product-level trends and forecasts, and identify high-value customers. The goal: Design and deliver an end-to-end business intelligence solution. The project will consist of 4 deliverables: * Connect and transform the raw data. * Build a relational data model. * Create new calculated columns and DAX measures. * Design an interactive report to analyze and visualize data. Important: The data is fictional and used for the purpose of demonstration. See the data on Kaggle.

Power BI is able to connect to data through database tables, flat files, cloud services, folders, etc, and create fully automated data shaping and loading ETL procedures. In this project. I will be using flat files stored locally and importing them into Power BI through the “Get data” feature.

Any data transformation that takes place will be done in the Power Query Editor. The feature of “Steps Applied” conveniently tracks all the changes made which can be automated for future use.

Creating data model: once we have transformed our data, let's build a data model that will connect multiple tables. Establishing relationships between tables is important when we query our data. To preserve data integrity and reduce redundancy we will make use of normalization. Models usually have two types of tables: data and lookup tables. Data tables contain values that help to create table relationships and Lookup tables provide descriptive, often text-based attributes

To add more depth to this analysis we need more metrics or quantitative values. So far, I have sales quantity and returns quantity. Power BI has Data Analysis Expressions or Dax that allows building new calculated columns or measures that add new dimensions to any analysis. "Measures" are similar to Excel pivot tables. For example, we could add a measure of the “ return rate” of a product and visualize which category of a product is returned more often or which region has more sales or returns.
